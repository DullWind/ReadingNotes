# 项目优化技术分析

> 说明：本文只整理优化思路、技术路径与工程取舍，不展开具体业务代码和业务规则。

## 1. 项目优化视角概览

这个项目属于大型 Unity 客户端中常见的 Lua 脚本层工程。优化重点不是单一函数提速，而是围绕“资源规模大、启动链路长、运行对象多、平台差异明显、热更新频繁”这些特点，形成一套端到端的工程优化体系。

可以把整体优化拆成六条主线：

1. 构建期优化：减少脚本体积、提升构建效率、让产物更适合差分更新。
2. 包体与更新优化：拆分首包和可选包，按需下载，降低首次安装和更新成本。
3. 启动速度优化：把必须阻塞的工作前置，把非关键工作延后，并用状态机控制风险。
4. 运行速度优化：减少单帧峰值，使用异步、分帧、LOD、裁剪和轻量对象替代。
5. 内存优化：减少短生命周期对象分配，控制资源生命周期，避免异步加载泄漏。
6. 监控闭环：采集 FPS、内存、Lua GC、网络延迟和资源加载行为，用数据驱动后续优化。

这些优化不是孤立存在的。例如，包体拆分会影响启动加载策略；资源按需下载会影响运行时加载失败处理；对象池能降低 GC，但也可能提高常驻内存。因此真正的优化目标不是“所有东西都更快”，而是在首包、启动、帧率、内存、网络和稳定性之间做平衡。

## 2. 包体优化思路

### 2.1 脚本预编译与统一打包

Lua 源码在发布前可以编译为中间格式，再统一打入数据包。这样做有几个收益：

- 减少源码文本体积。
- 降低运行时解析成本。
- 避免直接暴露完整脚本源码。
- 让脚本资源以统一格式参与版本管理、下载和加载。

这类方案适合脚本文件数量非常多的项目。它的核心思想是把“解释器启动后逐个解析文本脚本”的成本，尽量转移到构建阶段。

需要注意的是，预编译后的文件仍然要考虑平台兼容、Lua 版本兼容和调试便利性。发布包可以走预编译，开发环境则通常需要保留源码级调试能力。

### 2.2 多包拆分，而不是一个大包装到底

对于 WebGL 或弱网络环境，一个大包会显著拉长首屏时间。更合理的做法是按照加载优先级拆包：

- 首屏必要脚本和 UI 放入第一批包。
- 大地图、远景、低频系统放入后续包。
- 可选玩法、语音、视频、活动资源放入 DLC 或可选下载包。
- 资源绑定表、配置表等可按访问频率单独打包。

拆包的核心原则是：首包只放“没有它就进不了游戏”的东西。只要不是首进必需，就应该考虑延迟加载、后台下载或按需加载。

### 2.3 稳定构建产物，提升差分效率

更新包大小不仅取决于内容变化，也取决于构建产物是否稳定。如果每次构建都改变文件时间戳、压缩顺序或无关元数据，那么即使源码变化很少，差分包也会变大。

优化思路包括：

- 固定打包顺序。
- 固定文件时间戳。
- 避免随机内容写入产物。
- 将频繁变化与低频变化资源拆开。
- 大资源尽量独立打包，避免牵连更新。

这类优化对热更新项目尤其重要，因为用户感知到的是下载体积和等待时间，而不是构建系统内部改了多少文件。

### 2.4 移除无用配置和冗余协议文件

大型项目中配置表和协议文件容易持续膨胀。包体优化可以在构建期做裁剪：

- 移除客户端不需要的服务端字段。
- 移除当前平台或当前渠道不需要的配置。
- 将低频配置放入按需包。
- 对重复表、历史表、调试表做发布前过滤。

这类优化要非常谨慎，最好通过白名单或生成规则控制，避免误删运行时依赖。裁剪策略应该有验证流程，而不是人工猜测哪些文件“看起来没用”。

## 3. 内存优化思路

### 3.1 高频临时表复用

Lua 项目中最常见的内存压力来自短生命周期 table。比如事件参数、定时器参数、战斗消息、路径中间结果、UI 刷新数据等，都可能在一帧内大量创建和释放。

优化思路是建立 table pool：

- 使用时从池中取表。
- 用完后清理字段并放回池中。
- 对不同用途的表设置不同池上限。
- 周期性缩容，避免峰值过后仍常驻大量空表。
- Debug 环境检查是否残留 userdata 或重要引用。

收益是降低 Lua GC 频率和 GC 峰值，减少卡顿。风险是清理不彻底会造成脏数据和引用泄漏，因此必须有统一 reset 逻辑。

### 3.2 定时器回调参数池化

定时器系统如果频繁创建闭包和参数表，也会制造大量临时对象。优化方式是把回调数据结构池化：

- 注册定时器时从池中取 callback data。
- 执行完成或删除定时器时归还。
- 循环定时器保留数据，非循环定时器执行后释放。
- 支持按 tag 删除，防止场景切换后遗留回调。

这个思路适合 UI 动画、延迟销毁、网络超时、引导流程等大量使用定时器的客户端项目。

### 3.3 异步加载的废弃状态

异步加载有一个典型风险：请求发出后，对象所属页面或场景已经关闭，但资源稍后才返回。如果回调中继续创建对象，就会导致内存泄漏、空引用或无意义实例。

合理做法是给资源加载器设计状态机：

- 未加载。
- 加载中。
- 已加载。
- 已废弃。

当对象在加载中被销毁，不立即假设资源不存在，而是标记为“已废弃”。等异步回调返回后，如果发现状态已废弃，就立即释放返回资源，不再绑定到业务对象。

这个模式对大 UI、地图对象、特效、动态模型都很重要。

### 3.4 控制 Bundle 生命周期

Bundle 常见优化方向有两种：

- 保留依赖 Bundle，减少重复加载和加载抖动。
- 用完卸载 Bundle，降低常驻内存。

两者没有绝对优劣，取决于设备内存、资源复用率和加载频率。更成熟的做法是允许配置控制：

- 高配设备保留更多常用依赖，换取流畅度。
- 低配设备更积极卸载，换取内存安全。
- 大型共享资源和小型低频资源使用不同策略。
- 线上可以通过配置开关灰度调整。

### 3.5 预缓存与对象池的取舍

对象池和预缓存可以减少运行时首次加载卡顿，但会增加启动耗时和常驻内存。因此适合以下对象：

- 高频出现。
- 首次实例化成本高。
- 生命周期短且重复使用。
- 对帧率影响明显。

不适合无脑预加载所有资源。合理策略是先通过资源首次加载记录和性能埋点找出真正的卡点，再决定哪些对象进入池或预缓存列表。

## 4. 启动速度优化思路

### 4.1 用加载状态机管理启动流程

启动流程通常包含网络连接、账号登录、配置读取、资源初始化、地图数据、UI 初始化、SDK 初始化等步骤。如果这些逻辑散落在各处，很难定位慢点和异常。

使用加载状态机的收益是：

- 每一步职责清晰。
- 可以记录每个阶段耗时。
- 可以设置超时和错误处理。
- 可以区分首次启动和重连流程。
- 可以根据平台跳过不必要步骤。

这类状态机不是为了让代码看起来复杂，而是为了让启动链路可观测、可恢复、可灰度调整。

### 4.2 首屏只做必须完成的事

启动优化最重要的原则是区分“进游戏前必须完成”和“进游戏后可以继续完成”。

必须完成的通常包括：

- 关键配置读取。
- 账号和服务器握手。
- 首屏 UI 和基础资源。
- 必要地图或场景入口数据。

可以延后的通常包括：

- 商店、邮件、排行榜、活动等非首屏数据。
- 部分 SDK 扩展初始化。
- 可选资源检查。
- 非关键红点或推荐信息。
- 统计、诊断、后台预下载。

项目中常见做法是加载完成后设置额外任务队列，每隔一小段时间执行一批，避免首帧或首几秒堆积大量网络请求和初始化逻辑。

### 4.3 延迟初始化与分摊初始化

有些系统必须初始化，但不一定要在同一帧完成。可以把初始化拆成多个阶段：

- 基础数据结构初始化。
- 配置索引预计算。
- UI 缓存建立。
- 网络请求补齐。
- 资源预热。

然后按帧或按秒执行。这样总耗时可能没有减少，但用户感知的阻塞时间会明显下降。

### 4.4 Shader 预热策略

Shader 首次变体编译可能造成明显卡顿。常见优化是 Shader Warmup：

- 启动阶段预热关键变体。
- 加载界面期间按帧预热一部分。
- 记录已预热数量，下次启动继续利用历史进度。
- 对不同平台设置不同每帧预热上限。
- 允许线上开关关闭或调整预热策略。

预热的关键是控制节奏。一次性预热太多会拖慢启动；完全不预热则可能在战斗或地图交互中卡顿。按帧预热是一种折中。

### 4.5 WebGL 首包特殊处理

WebGL 对首包大小、网络延迟、浏览器缓存和内存限制更敏感，因此通常需要独立策略：

- 首包拆得更细。
- 隐藏或简化部分加载界面资源。
- 下载队列串行化，避免并发抢占内存和带宽。
- 本地缓存命中时跳过重复下载。
- 资源准备完成后再进入正式流程。

WebGL 优化的目标不是让所有资源都更早加载，而是让用户尽早进入可交互状态。

## 5. 运行速度优化思路

### 5.1 异步加载和逐帧实例化

大量对象同时实例化会造成主线程卡顿。优化思路是：

- 资源加载尽量异步。
- 实例化支持逐帧执行。
- 大量地图对象进入构建队列。
- 每帧设置时间预算或数量预算。
- 优先构建屏幕内、玩家相关、交互相关对象。

这个方法不会减少总工作量，但会把峰值摊平，避免用户看到明显卡顿。

### 5.2 世界对象构建队列

大地图或多人同屏场景中，服务器可能一次推送大量对象变化。如果客户端收到后立即全部构建，会造成瞬时卡顿。

更合理的做法是建立待构建队列：

- 收到对象数据后先放入队列。
- 同一个对象的新消息覆盖旧消息，避免处理过期数据。
- 每帧根据预算处理一部分。
- 重要对象可优先处理。
- 对象删除时同步移除待构建项。

这个思路特别适合地图对象、战斗单位、远景标记、动态建筑等大规模对象场景。

### 5.3 LOD 和战略视图

LOD 的核心思想是：用户看不清细节时，就不要渲染细节。

在远景或战略视图中，可以把复杂对象替换为轻量图标、简化模型或合批对象：

- 近景使用完整模型和特效。
- 中景减少动画、材质和子节点。
- 远景使用低模或图标。
- 超出视野直接销毁或隐藏。
- 不同视图等级向服务器请求不同粒度数据。

这类优化通常收益巨大，因为地图对象和单位数量往往远高于普通 UI 对象。

### 5.4 视野裁剪和对象销毁

当对象离开当前视野，继续保留完整 GameObject 会浪费 CPU、内存和渲染资源。常用策略：

- 战术视图下保留近处对象。
- 战略视图下保留必要图标。
- 离屏对象销毁完整实体，只保留逻辑数据。
- 回到视野时再异步重建。
- 对副本、演出、引导等特殊场景设置例外。

裁剪策略必须小心处理状态同步。对象销毁的是表现层，不应该丢失逻辑数据。

### 5.5 特效裁剪和简化表现

战斗特效、飘字、血条、范围提示等是性能热点。优化思路包括：

- 远景跳过复杂特效。
- 战略视图只显示必要结果，不播放完整过程。
- 只给玩家相关目标显示高优先级表现。
- 对低帧率或低画质设置启用简化模式。
- 控制同屏特效数量和实例化频率。

这类优化需要关注体验边界：不能因为裁剪导致玩家无法理解战斗结果或关键反馈。

### 5.6 轻量渲染对象替代完整 GameObject

当某类对象数量非常大，而表现又比较统一时，可以用更轻量的渲染路径替代完整 prefab。比如统一图标、低模、合批实例或原生渲染对象。

收益是：

- 减少 GameObject 数量。
- 减少 Transform 层级开销。
- 减少 Lua 与 C# 交互频率。
- 更适合大规模同屏对象。

风险是灵活性下降，因此适合远景、战略图标、重复单位等表现规则相对固定的场景。

## 6. 下载与更新优化思路

### 6.1 可选 DLC

可选 DLC 的目标是把“不是所有玩家都立即需要”的资源从主包中拆出去。常见对象包括：

- 低频玩法资源。
- 大型语音、视频、皮肤资源。
- 活动资源。
- 平台或渠道专属资源。

下载策略可以分为：

- 首次进入后后台下载。
- 进入相关功能前提示下载。
- Wi-Fi 条件下预下载。
- 根据用户进度解锁下载。

可选 DLC 的难点不是下载，而是状态管理：本地是否存在、版本是否匹配、CRC 是否正确、失败如何重试、下载中是否允许进入功能等。

### 6.2 差分补丁

完整下载文件简单但成本高。差分补丁通过比较本地旧文件和服务器新文件，只下载变化部分。

典型流程是：

1. 读取本地文件版本、大小和 CRC。
2. 查询服务器是否存在对应旧版本的差分包。
3. 如果存在且差分包更小，则下载差分包。
4. 下载后校验差分包大小和 CRC。
5. 使用 patch 工具生成新文件。
6. 校验新文件大小和 CRC。
7. 成功后替换本地文件，失败则回退完整包下载。

这个方案能显著降低热更新体积，但必须把校验做完整，否则会产生难以排查的资源损坏问题。

### 6.3 下载队列和互斥

客户端中可能同时存在主更新、可选 DLC、活动资源、WebGL 资源等多种下载。如果完全并发，会抢占带宽、增加内存压力，并让失败处理复杂。

更稳妥的做法是下载队列化：

- 同一时间只处理有限数量任务。
- 关键下载优先级高于后台下载。
- 不同下载系统之间设置互斥。
- 每个任务有超时、进度回调和失败回调。
- 失败后切换 CDN 或延迟重试。

下载优化关注的不只是速度，也包括稳定性、恢复能力和用户可理解的提示。

### 6.4 本地缓存与版本记录

下载完成后应保存本地文件信息，包括文件名、版本、大小、CRC、所属 DLC 等。下次启动时先检查本地记录，再决定是否需要访问网络。

缓存策略要防止两类问题：

- 记录存在但文件损坏。
- 文件存在但记录过期。

因此文件大小和 CRC 校验仍然是必要的，尤其是在移动端弱存储、异常退出或 WebGL 缓存环境下。

## 7. 性能监控闭环

### 7.1 FPS 分位数比平均值更有价值

平均 FPS 容易掩盖卡顿。更有效的指标包括：

- 5% low FPS。
- 10% low FPS。
- 卡顿帧数量。
- 场景切换时 FPS。
- 同屏对象数量对应的 FPS。
- 不同设备档位的 FPS。

用分位数能更接近用户体感，因为玩家对偶发卡顿非常敏感。

### 7.2 内存和 Lua GC 监控

大型 Lua 项目需要同时关注 Unity 内存和 Lua 内存：

- 总内存使用。
- Lua `collectgarbage("count")`。
- GC 发生时间。
- 卡顿时的内存快照。
- 资源加载历史。

如果 FPS 下降同时 Lua 内存快速增长，通常说明临时表、闭包或缓存没有被正确释放。如果 Unity 内存上涨，则更可能是资源、贴图、Bundle 或 GameObject 生命周期问题。

### 7.3 资源首次加载记录

首次加载某个资源时记录下来，可以帮助定位卡顿来源：

- 哪些资源在游戏中途首次加载。
- 哪些资源应该预热但没有预热。
- 哪些资源被频繁重复加载。
- 哪些 UI 打开时触发大量资源加载。

这类数据可以反向指导对象池、预缓存和拆包策略，而不是靠经验决定。

### 7.4 Lua Profiler

Lua profiler 可以帮助定位 CPU 热点：

- 哪些函数累计耗时高。
- 哪些函数调用次数异常多。
- 哪些调用栈导致卡顿。
- 某个场景或操作下的主要耗时路径。

Profiler 不适合长期在线开启，因为 hook 本身有开销。更适合作为调试工具、灰度诊断工具或特定 Debug 配置下使用。

## 8. 风险与改进建议

### 8.1 不要过度预加载

预加载能减少运行时卡顿，但也会增加启动时间和内存。建议只预加载经过数据证明的高频高成本资源。

### 8.2 对象池必须可控

对象池不是越大越好。池上限、缩容策略、Debug 校验和生命周期规则都必须清晰。否则对象池会从优化手段变成内存泄漏来源。

### 8.3 拆包粒度要稳定

拆包太粗会影响首包和更新包；拆包太细会增加请求数量、管理复杂度和失败概率。合理粒度应该结合加载顺序、资源复用率和 CDN 表现决定。

### 8.4 差分更新必须有回退

差分 patch 链路任何一步失败，都应该能回退到完整文件下载。否则用户可能陷入资源损坏但无法恢复的状态。

### 8.5 运行时优化要保留体验底线

LOD、裁剪、特效简化都不能破坏关键反馈。建议把表现分成“必须显示”“可简化”“可隐藏”三类，而不是简单按距离一刀切。

### 8.6 建议补充 Unity 工程层分析

当前分析基于 Lua 层和构建脚本。若要继续深入，还应补充 Unity 工程层内容：

- AssetBundle 构建规则。
- 贴图压缩格式。
- Shader Variant 收集规则。
- Addressables 或自研资源系统实现。
- C# 层对象池和渲染优化。
- IL2CPP、裁剪、托管内存策略。

## 9. 总结

这个项目采用的是一套面向大型客户端的综合优化体系。核心不是某个单点技巧，而是把优化前移到构建期、拆分到资源更新期、分摊到启动和运行期，并通过监控数据持续验证。

最值得总结的优化原则是：

1. 构建期做能提前做的事，减少运行时负担。
2. 首包只放必要内容，其余资源按需或后台下载。
3. 启动链路状态机化，非关键任务延迟执行。
4. 运行期避免单帧峰值，用异步、分帧和队列摊平压力。
5. 大规模地图对象使用 LOD、视野裁剪和轻量表现。
6. 高频临时数据用池化减少 Lua GC。
7. 下载更新必须有 CRC、差分和失败回退机制。
8. 所有优化都需要 FPS、内存和资源加载数据闭环验证。

### 10.3 可选 DLC 与差分更新

```pseudo
function CheckOptionalPackage(packageInfo):
    localInfo = LoadLocalPackageInfo(packageInfo.name)

    if localInfo exists and VerifyFile(localInfo.path, packageInfo.size, packageInfo.crc):
        RegisterPackage(localInfo.path)
        return "ready"

    diffInfo = QueryDiffInfo(
        packageName = packageInfo.name,
        fromCrc = localInfo.crc,
        toCrc = packageInfo.crc
    )

    if diffInfo exists and diffInfo.size < packageInfo.size:
        QueueDownload(diffInfo)
        return "need_diff_download"

    QueueDownload(packageInfo)
    return "need_full_download"

function OnDownloadFinished(downloadedFile, targetPackage):
    if downloadedFile.isDiff:
        if not VerifyFile(downloadedFile.path, downloadedFile.size, downloadedFile.crc):
            QueueDownload(targetPackage.fullFile)
            return

        patchedFile = ApplyPatch(oldFile, downloadedFile)
        if not VerifyFile(patchedFile, targetPackage.size, targetPackage.crc):
            QueueDownload(targetPackage.fullFile)
            return

        SaveLocalPackageInfo(patchedFile)
    else:
        if not VerifyFile(downloadedFile.path, targetPackage.size, targetPackage.crc):
            RetryOrSwitchCDN(targetPackage)
            return

        SaveLocalPackageInfo(downloadedFile)

    RegisterPackage(targetPackage)
```

关键点：

- 差分不是替代完整下载，而是优先尝试的优化路径。
- 每一步都需要大小和 CRC 校验。
- 差分失败必须能回退完整包。

### 10.4 下载队列与互斥调度

```pseudo
DownloadQueue = Queue()
CurrentDownload = null

function AddDownloadTask(task):
    if IsAlreadyDownloaded(task):
        return
    DownloadQueue.Enqueue(task, priority = task.priority)

function UpdateDownloadManager(deltaTime):
    if CurrentDownload != null:
        CheckTimeout(CurrentDownload)
        return

    if IsOtherDownloadSystemBusy():
        return

    task = DownloadQueue.DequeueHighestPriority()
    if task == null:
        return

    CurrentDownload = task
    StartHttpDownload(
        url = SelectCDN(task),
        timeout = task.timeout,
        onProgress = UpdateProgress,
        onFinish = OnDownloadFinished
    )

function OnDownloadFinished(result):
    task = CurrentDownload
    CurrentDownload = null

    if result.success:
        ValidateAndRegister(task, result.file)
    else:
        task.retryCount += 1
        if task.retryCount <= task.maxRetry:
            task.cdnIndex = NextCDN(task.cdnIndex)
            AddDownloadTask(task)
        else:
            ReportDownloadFailure(task, result.error)
```

关键点：

- 控制并发可以降低内存和网络抖动。
- 不同下载系统之间需要互斥或优先级。
- 失败后切 CDN、延迟重试、最终上报。

### 10.8 Table 池

```pseudo
TablePools = Map<PoolName, Stack<Table>>()
PoolLimits = Map<PoolName, Number>()
LastUseTime = Map<PoolName, Time>()

function AcquireTable(poolName):
    pool = TablePools[poolName]
    LastUseTime[poolName] = Now()

    if pool is not empty:
        return pool.Pop()

    return NewTable()

function ReleaseTable(poolName, table):
    ResetTableFields(table)

    pool = TablePools[poolName]
    if pool.Count >= PoolLimits[poolName]:
        return

    pool.Push(table)

function UpdateTablePools():
    every 60 seconds:
        for each poolName in TablePools:
            if Now() - LastUseTime[poolName] > 1 second:
                ShrinkPool(poolName, ratio = 0.3)
```

关键点：

- 池化适合高频、短生命周期 table。
- 归还前必须清理引用。
- 池需要上限和缩容。

### 10.9 定时器回调数据复用

```pseudo
CallbackPool = ObjectPool()
ActiveCallbacks = Map<TimerId, CallbackData>()

function RegisterTimer(delay, callback, params, repeatInterval = null, tag = null):
    if delay <= 0 and repeatInterval == null:
        callback(params)
        return null

    data = CallbackPool.Acquire()
    data.callback = callback
    data.params = params
    data.repeatInterval = repeatInterval
    data.tag = tag

    timerId = NativeTimer.Add(delay, repeatInterval, OnTimerCallback)
    ActiveCallbacks[timerId] = data
    return timerId

function OnTimerCallback(timerId):
    data = ActiveCallbacks[timerId]
    if data == null:
        return

    shouldContinue = data.callback(data.params)
    if not shouldContinue or data.repeatInterval == null:
        ActiveCallbacks.Remove(timerId)
        data.Reset()
        CallbackPool.Release(data)

function DeleteTimer(timerId):
    NativeTimer.Delete(timerId)
    data = ActiveCallbacks.Remove(timerId)
    if data != null:
        data.Reset()
        CallbackPool.Release(data)
```

关键点：

- 减少定时器参数表和闭包造成的 GC 压力。
- 场景切换时应支持按 tag 批量删除。
- 删除定时器和执行完成都要回收 callback data。

### 10.10 异步资源加载的废弃状态

```pseudo
enum LoadState:
    None
    Loading
    Loaded
    Discarded

class ResourceHandle:
    state = None
    object = null

    function LoadAsync(assetName, onLoaded):
        if state == Loaded:
            onLoaded(object)
            return

        if state == Loading:
            return

        state = Loading
        AsyncLoad(assetName, function(result):
            if state == Discarded:
                DestroyResource(result)
                state = None
                return

            state = Loaded
            object = result
            onLoaded(object)
        )

    function Destroy():
        if state == Loading:
            state = Discarded
            return

        if state == Loaded:
            DestroyResource(object)
            object = null
            state = None
```

关键点：

- 异步回调返回时，拥有者可能已经销毁。
- `Discarded` 状态能避免泄漏和无效绑定。
- 销毁逻辑要同时处理“已加载”和“加载中”。

### 10.11 世界对象构建队列

```pseudo
BuildQueue = Queue<ObjectId>()
PendingObjects = Map<ObjectId, PendingObjectData>()

function OnObjectMessage(message):
    objectId = message.objectId

    if PendingObjects contains objectId:
        old = PendingObjects[objectId]
        if old.timestamp > message.timestamp:
            return
        PendingObjects[objectId] = MergeOrReplace(old, message)
    else:
        PendingObjects[objectId] = CreatePendingData(message)
        BuildQueue.Enqueue(objectId)

function UpdateObjectBuilder():
    startTime = Now()
    processed = 0

    while BuildQueue is not empty:
        if Now() - startTime > FrameBuildBudget:
            break
        if processed > MaxBuildCountPerFrame:
            break

        objectId = BuildQueue.Dequeue()
        pending = PendingObjects.Remove(objectId)
        if pending == null:
            continue

        BuildOrUpdateViewObject(pending)
        processed += 1

function OnObjectRemoved(objectId):
    PendingObjects.Remove(objectId)
    DestroyViewObject(objectId)
```

关键点：

- 消息先入队，不立即全量构建。
- 同一对象的新消息覆盖旧消息。
- 每帧受时间预算和数量预算限制。

### 10.16 优化决策流程

```pseudo
function OptimizePerformanceProblem(problem):
    data = CollectMetrics(problem)

    if data.firstLoadResourceSpike:
        ConsiderPreloadOrPackageSplit(data.resources)

    if data.luaMemoryGrowing:
        FindHighFrequencyAllocations()
        AddTablePoolOrFixReferences()

    if data.frameSpikeFromInstantiation:
        MoveToAsyncOrFrameBudgetQueue()

    if data.renderCostHigh:
        AddLODOrEffectCulling()

    if data.startupTooSlow:
        MoveNonCriticalTasksToPostLoading()
        SplitStartupStepsAndAddTimeouts()

    if data.updateDownloadTooLarge:
        ImprovePackageSplitAndDiffPatch()

    VerifyWithMetricsBeforeAndAfter()
```

关键点：

- 先看数据，再选手段。
- 每个优化都要验证收益和副作用。
- 优化目标要明确：是降首包、降启动、降卡顿，还是降内存。
