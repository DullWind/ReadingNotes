# BM25 + 向量 + 图检索与 RRF 融合

## 一句话总结

这是一种多路混合检索方案：

```text
BM25 找“字面匹配”
向量检索找“语义相似”
图检索找“实体关系”
RRF 根据各路排名合并结果
```

三种检索方式负责不同类型的相关性，RRF（Reciprocal Rank Fusion，倒数排名融合）则把它们组合成一个稳定的最终排序。该方案常用于企业知识库、RAG、代码搜索和复杂问答。

> 本文中的“图检索”主要指基于知识图谱、实体关系图或文档关系图的检索，而不是图像搜索。

***

## 一、为什么要组合三种检索

任何一种单独的检索方式都有明显盲区。

| 检索方式 | 判断相关性的依据 | 擅长 | 主要盲区 |
| --- | --- | --- | --- |
| BM25 | 关键词、词频、逆文档频率 | 精确术语、错误码、产品型号、人名 | 难以处理同义词和语义改写 |
| 向量检索 | 文本在向量空间中的语义距离 | 自然语言问题、同义表达、概念匹配 | 精确数字和罕见术语可能不稳定 |
| 图检索 | 实体、关系、路径和邻居 | 依赖、归属、因果、多跳关系 | 依赖图谱的完整性和准确性 |

例如用户搜索：

```text
登录失败和 Redis 有什么关系？
```

三路检索可能分别得到：

- BM25：找到同时出现“登录失败”和“Redis”的故障文档。
- 向量检索：找到“缓存不可用导致认证异常”的文档，即使没有使用原问题中的词。
- 图检索：沿着“登录服务 -> 会话服务 -> Redis”的关系找到关联组件和对应文档。

组合之后，系统既不会错过精确关键词，也能理解语义改写和实体之间的结构化关系。

***

## 二、BM25：寻找字面匹配

BM25 是经典的稀疏检索算法，可以理解为经过改进的关键词匹配。

它主要关注：

- 查询词是否出现在文档中。
- 查询词在文档中出现多少次。
- 查询词在整个语料库中是否稀有。
- 文档长度，避免长文档仅凭词多获得过高分数。

BM25 特别适合：

```text
ERR_CONNECTION_RESET
RedisClusterTimeout
订单号 10458
API 名称、类名、函数名
产品型号和专业术语
```

这些内容通常要求精确匹配，只靠向量检索可能会召回“意思接近但并非同一个对象”的结果。

***

## 三、向量检索：寻找语义相似

向量检索先用 Embedding 模型把问题和文档转换成向量，再根据余弦相似度、内积或欧氏距离查找相近内容。

例如下面几句话字面不同，但语义接近：

```text
用户无法登录
认证过程失败
账号进不去系统
登录接口返回异常
```

向量检索可以把它们映射到相近的语义空间，因此适合处理：

- 同义词。
- 口语化问题。
- 查询改写。
- 没有共享关键词但概念相关的文本。

它的主要问题是精确性不总是可靠。例如两个不同错误码、版本号或产品型号，在语义上可能很接近，但业务含义完全不同。

***

## 四、图检索：寻找实体关系

图检索把知识表示为节点和边：

```text
节点：服务、接口、人员、文档、故障、数据库
边：依赖、调用、属于、负责、导致、引用
```

例如：

```text
登录服务 --调用--> 会话服务 --依赖--> Redis
    |
    +--关联文档--> 登录故障处理手册
```

当用户查询“登录失败和 Redis 有什么关系”时，系统可以：

1. 从问题中识别“登录”和“Redis”实体。
2. 在图中查找它们之间的路径。
3. 找到路径上的服务、故障和关联文档。
4. 把关联文档转换成可以参加融合排序的候选文本块。

图检索常见的策略包括：

- 实体邻居扩展：查找目标实体一跳或两跳内的节点。
- 路径检索：查找两个实体之间的关系路径。
- 子图检索：返回与问题相关的一组实体和关系。
- 图算法：使用 PageRank、个性化 PageRank、社区发现等方式寻找重要节点。
- GraphRAG：先在图或社区摘要中定位相关区域，再回到原始文本取证。

图检索的价值不只是“多召回一些文档”，而是显式表达文档中不容易通过局部文字匹配得到的关系。

***

## 五、为什么不能直接相加三路分数

三种系统使用的分数不在同一个尺度上。例如：

```text
BM25 分数：12.7
向量相似度：0.83
图路径分数：2.4
```

这些数字的定义和分布完全不同。直接相加会让数值范围较大的检索方式天然占优势。

虽然可以分别做归一化，但归一化通常受到以下因素影响：

- 查询不同，分数分布也不同。
- 索引或模型升级会改变分数范围。
- 图检索可能根本没有统一的相似度定义。
- 极端值容易影响 Min-Max 等归一化方法。

RRF 的思路是跳过原始分数，只使用每个结果在各路检索中的名次。

***

## 六、RRF 的工作原理

普通 RRF 的公式是：

```text
RRF(d) = sum(1 / (k + rank_i(d)))
```

加权 RRF 可以写成：

```text
RRF(d) = sum(w_i / (k + rank_i(d)))
```

其中：

- `d`：某篇候选文档或文本块。
- `rank_i(d)`：它在第 `i` 路检索中的排名，从 1 开始。
- `w_i`：该路检索的权重，普通 RRF 中都为 1。
- `k`：平滑参数，常见初始值为 60。

假设文档 A 的排名是：

| 检索方式 | 排名 | `k=60` 时的贡献 |
| --- | ---: | ---: |
| BM25 | 1 | `1 / 61` |
| 向量检索 | 3 | `1 / 63` |
| 图检索 | 2 | `1 / 62` |

最终分数为：

```text
RRF(A) = 1/61 + 1/63 + 1/62 ≈ 0.04839
```

一篇文档如果同时被多路检索排在前面，会获得较高的累计分数。只在一路中排名很高的结果仍有机会进入最终列表，但通常不如多路共同认可的结果稳定。

### 参数 k 的含义

- `k` 较小：更强调每路最前面的少数结果，排名差异影响较大。
- `k` 较大：排名贡献更平滑，多路共同命中变得更重要。

`60` 是常见起点，但不是固定标准，应根据真实查询集评测。

### 加权 RRF

如果业务更依赖某一路检索，可以加入权重。例如故障知识库可能设置：

```text
Score(d) =
    1.2 / (60 + rank_BM25(d))
  + 1.0 / (60 + rank_vector(d))
  + 0.8 / (60 + rank_graph(d))
```

这表示系统稍微更信任错误码、组件名等精确关键词。权重应该通过评测数据确定，而不是只凭直觉设置。

## 七、完整检索流程

```text
用户问题
   |
   +-- BM25 检索 ---------------- Top N
   |
   +-- Embedding -> 向量检索 ----- Top N
   |
   +-- 实体识别 -> 图检索 -------- Top N
                         |
                         v
                 映射成统一文档 ID
                         |
                         v
                    去重与权限过滤
                         |
                         v
                      RRF 融合
                         |
                         v
             可选：Cross-Encoder/LLM 重排
                         |
                         v
                 Top K 文本交给 LLM
```

伪代码：

```python
bm25_results = bm25.search(query, top_n=50)
vector_results = vector_store.search(embed(query), top_n=50)
graph_results = graph_retriever.search(query, top_n=50)

rank_lists = [
    normalize_to_chunk_ids(bm25_results),
    normalize_to_chunk_ids(vector_results),
    normalize_to_chunk_ids(graph_results),
]

scores = {}
k = 60

for results in rank_lists:
    for rank, chunk_id in enumerate(results, start=1):
        scores[chunk_id] = scores.get(chunk_id, 0) + 1 / (k + rank)

fused = sorted(scores, key=scores.get, reverse=True)
final_results = rerank(query, fused[:30])[:8]
```

## 八、工程实现的关键问题

### 1. 统一结果粒度

RRF 融合的对象必须具有统一身份，例如全部映射成：

```text
chunk_id
document_id
passage_id
```

不能让 BM25 返回文本块、向量检索返回整篇文档、图检索返回实体节点，然后直接进行排名融合。

如果最终问答以文本块为依据，图检索需要完成如下映射：

```text
图节点或路径 -> 关联文档 -> 关联文本块 chunk_id
```

### 2. 去重与父文档聚合

同一篇文档可能有多个相邻文本块进入候选列表。常见处理方式：

- 按 `chunk_id` 去重。
- 限制每篇文档最多进入几个文本块。
- 先在块级融合，再按 `document_id` 聚合。
- 合并相邻块，避免给 LLM 提供重复上下文。

### 3. 限制图扩展范围

公共服务、核心数据库等枢纽节点可能连接大量文档。如果不限制，会引入许多弱相关结果。

常见限制包括：

- 最大跳数，例如 1～2 跳。
- 允许使用的关系类型。
- 路径长度惩罚。
- 每个节点最多扩展多少邻居。
- 时间、租户、部门和权限过滤。
- 对超高出度节点降权。

### 4. 在正确阶段执行权限过滤

企业检索必须保证用户无权访问的文档不会通过图关系或向量索引泄露。权限过滤最好下推到各路召回阶段，并在融合前后再次校验。

### 5. RRF 后增加精排

RRF 擅长稳定地合并候选集，但它只看排名，不深入判断问题与文档的细粒度相关性。

更完整的流程通常是：

```text
多路召回 -> RRF 得到 20～100 个候选 -> Reranker 精排 -> 取前 5～10 个
```

精排器可以使用 Cross-Encoder，也可以使用专门设计的 LLM 相关性判断器。

## 九、RRF 的优点与限制

### 优点

- 不需要比较不同检索系统的原始分数。
- 算法简单、稳定、容易实现。
- 对某一路偶然出现的异常高分不敏感。
- 能奖励被多路检索共同命中的结果。
- 新增或移除一路召回器的成本较低。

### 限制

- 丢弃了原始分数中的“领先幅度”。
- 排名第 1 的弱匹配与排名第 1 的强匹配，在该路中的贡献相同。
- 各路召回数量过少时，可能遗漏后续本可融合上升的结果。
- 如果某一路质量很差，它仍会给噪声结果加分。
- 图检索返回的粒度不一致时，融合结果容易失真。
- RRF 不能替代最终的相关性精排和事实验证。

***

## 十、适合与不适合的场景

### 适合

- 查询中同时包含自然语言和精确术语。
- 企业知识库中存在服务、人员、文档、项目等关系数据。
- 问题需要理解“谁依赖谁”“故障影响哪些业务”等多跳关系。
- 单独使用向量检索经常召回语义相似但事实不相关的内容。
- 搜索系统需要兼顾召回率、精确率和结果稳定性。

### 不一定需要

- 数据量很小，简单关键词已经足够。
- 没有可靠的实体关系数据，图谱维护成本高于收益。
- 查询只要求严格匹配编号、名称或固定字段。
- 延迟预算极低，无法同时运行多路召回。

图检索不是必选项。常见演进路径是：

```text
BM25
  -> BM25 + 向量
  -> BM25 + 向量 + RRF
  -> 有明确关系检索需求后再加入图检索
```

## 十一、参数与评测建议

可以先使用以下参数作为基线，再通过数据调整：

```text
BM25 Top N：50
向量检索 Top N：50
图检索 Top N：20～50
RRF k：60
RRF 后保留：20～50
Reranker 后保留：5～10
图扩展深度：1～2 跳
```

评测时不要只看最终回答是否“感觉正确”，应分别观察：

- Recall@K：正确证据是否进入候选集。
- MRR：第一个正确结果出现得有多靠前。
- NDCG@K：整体排序质量。
- Hit Rate：前 K 个结果中是否包含正确结果。
- 最终回答正确率和引用准确率。
- 检索延迟、Token 数量和系统成本。

还应该对不同类型的查询单独评测：

```text
精确术语类
语义改写类
实体关系类
多跳推理类
时间敏感类
无答案类
```

这样才能判断 BM25、向量和图检索各自提供了多少真实增益。

***

## 最终理解

这套方案的核心不是让三种检索互相竞争，而是让它们分工协作：

```text
BM25 保证字面上的准确
向量检索扩大语义召回
图检索补充结构化关系
RRF 消除分数尺度差异并融合排名
Reranker 负责最后的细粒度判断
```

如果只记住一个原则，就是：

> 先让不同检索器从不同角度找到候选，再把候选统一到同一种文档粒度，用 RRF 做稳定融合，最后用精排模型挑出真正应该交给 LLM 的证据。
